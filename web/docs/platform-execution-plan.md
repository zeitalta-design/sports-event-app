# データプラットフォーム 実行計画

## 1. 開発方針

「共通基盤を先に固め、優先案件から順に横展開」する。
8案件を同じ深さで同時開発しない。レイヤーを分けて段階的に進める。

### 原則
- 過剰抽象化しない。まず動くMVPを優先
- 共通化は基盤レイヤーまで。ドメイン固有詳細は別テーブル・別モジュール
- 保留案件のロジックは core に混ぜない
- 既存資産を最大限活用するが、スポーツ/比較メディア前提のロジックは外す

---

## 2. レイヤー構成

### Layer 1: 共通基盤（最優先）
全ドメインが依存する基盤機能。先行して整備する。

| 責務 | 既存状況 | 対応 |
|---|---|---|
| 事業者/法人エンティティ管理 | 各ドメインに個別テーブル | 共通 organizations テーブル検討 |
| ソース管理 | data_sources テーブルあり | 拡張 |
| クロール/取込ログ | sync_runs テーブルあり | そのまま利用 |
| タグ | item_tags あり（SaaS用） | 汎用化 |
| 差分検知 | change_detector.js あり | そのまま利用 |
| 通知基盤 | notifier.js あり | 拡張 |
| 検索基盤 | 各ドメイン個別実装 | 横断検索 API あり、拡充 |
| 管理画面 | AdminNav + 各ドメイン管理あり | 共通フレーム維持 |
| 重複判定 | なし | 新規実装 |
| 手動修正UI | admin CRUD あり | そのまま利用 |
| 監視ジョブ共通枠組み | sync-runner.js + cron-sync.js | そのまま利用 |

### Layer 2: 先行立ち上げ案件（Phase 1）

2. **許認可検索** — 既存実装あり、拡充

### Layer 3: 横展開案件（Phase 2-3）

## 4. 流用ルール

### 積極流用してよいもの
- 認証・権限管理（auth.js, admin-api-guard.js）
- お気に入り（item-favorites API パターン）
- 保存検索（saved-searches パターン）
- 通知基盤（notifier.js）
- 一覧/詳細ページ骨組み（DomainListPage, DomainDetailPage）
- 管理画面フレームワーク（AdminNav + admin CRUD パターン）
- CSV取込基本部品（importers パターン）
- タグ管理（item_tags）
- 差分監視（change-detector.js, sync-runner.js）
- adapter/scraper 枠組み（automation/adapters, sources）
- ドメインレジストリ（domain-registry.js, registerDomain）

### 流用注意（ドメインに応じて判断）
- 比較UI — 行政処分系では不要
- ランキング — そのままでは使えない場合が多い
- おすすめロジック — ドメイン固有
- スコアリング — ドメイン固有
- SEOテンプレート — 文言をドメインに合わせる
- ドメイン固有フィルタ — 各 config で定義
- スポーツ/比較メディア寄り文言 — 外す

---

## 5. 保留案件の扱い

- 既存コードはそのまま維持（削除しない）
- 保留案件のドメイン固有ロジックを共通基盤に混ぜない
- 共通基盤の改善が保留案件にも波及するのはOK
- 保留案件のデータは /platform/dashboard に引き続き表示（件数0でも問題なし）
